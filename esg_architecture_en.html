2 — Mapping table (Framework Mapping)
⇄
framework_mapping — Adding a framework = adding rows here, nothing else
Central pivot table. Each row links a framework requirement to a canonical_id. A single canonical_id can have N rows — one per framework.
canonical_id FK framework VARCHAR requirement_code VARCHAR section VARCHAR question_label TEXT mandatory BOOLEAN cardinality ENUM(exact|partial|divergent) answer_format ENUM(number|text|boolean|select) unit VARCHAR confidence_level ENUM(high|medium|low) notes TEXT valid_from DATE valid_to DATE
canonical_id (UUID)
3 — Canonical model — The SaaS universal language
◎
Canonical Requirement Model — Independent of any framework · Stable over time
The immutable core of the system. These tables do not change when a new framework is added. canonical_ids are the single source of truth.
canonical_requirement TABLE
PK canonical_id UUID
slug VARCHAR — human-readable
domain ENUM(E,S,G,X)
topic VARCHAR
concept VARCHAR
data_type ENUM(number,text,bool,select)
unit VARCHAR nullable
is_quantitative BOOLEAN
label_fr TEXT
label_en TEXT
description TEXT
FK taxonomy_id UUID
canonical_taxonomy TABLE
PK taxonomy_id UUID
domain ENUM(E,S,G,X)
topic VARCHAR
sub_topic VARCHAR
concept_slug VARCHAR — unique
FK parent_id UUID nullable (self)
depth_level INT
is_leaf BOOLEAN

4 — Render layer (Framework Renderer)
⊞
framework_renderer — Transforms a canonical datapoint into a framework-specific response
Service/function that joins canonical_datapoint + framework_mapping to produce the response in the format and nomenclature of each framework. Handles transformation logic, unit validation, and narrative response formatting.
input: canonical_id + org_id + year input: target_framework output: requirement_code output: section output: formatted_value output: validation_status output: completeness_score

6 — Correspondence types (cardinality)
Exact — 1:1
Perfect match
Same concept, same unit, same scope across all frameworks. One single field feeds everything.
GHG_SCOPE1_DIRECT_tCO2e
→ ESRS E1-6 · CDP C6.1 · GRI 305-1 · IFRS S2.29a
Partial — 1:N
Same concept, different granularity
Multiple values for the same canonical_id depending on the framework. E.g. Scope 2 in two calculation methods.
GHG_SCOPE2_tCO2e
→ value_market (CSRD, CDP)
→ value_location (CSRD, CDP, GRI)
Divergent — N:1
Similar concept, distinct definition
Frameworks do not define exactly the same thing. The notes field documents the nuance and guides the user.
SUPPLY_CHAIN_SOCIAL_COVERAGE
→ ESRS S2 (due diligence)
→ GRI 413 (community impact)
Absent — framework-only
Specific to a single framework
Some requirements only exist in one framework. The canonical_id is created but without mapping to others.
TAX_COUNTRY_BY_COUNTRY
→ GRI 207 only
(no CSRD or CDP equivalent)
