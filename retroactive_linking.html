4Outcome States — Three possible results after assessment
✅
Full match — link immediately
All required template columns are present in the existing DS. No schema change needed. The link is created instantly with no user action required beyond confirmation.
coverage: 100% · missing: 0 · action: link_only
⚠️
Partial match — resolve then link
Some columns match, some are missing or ambiguous. User goes through the Column Resolver to confirm aliases and acknowledge added columns. Most common scenario.
coverage: 40–99% · action: resolve → add → link
🚫
No match — block linking
The existing DS is structurally incompatible — wrong domain, wrong data type for the key metric. System explains why and suggests either creating a new DS from the template or choosing a different canonical_id.
coverage: <30% · action: block + suggest alternative
Core design principles for retroactive linking
🔒
Never destructive
Existing columns are never removed, renamed, or retyped. The DS schema only ever grows — it never shrinks. Anything already connected to those columns keeps working.
🗺
Mapping over migration
The ds_column_mapping table is the bridge. The renderer reads original column names through the mapping — the DS data itself is never touched or moved.
🤝
System suggests, user confirms
Auto-mapping runs first on name similarity and type compatibility. Ambiguous matches (medium confidence) always require explicit user confirmation before being saved.
📋
Full audit trail
Every mapping decision is recorded: who confirmed it, when, with what confidence score. This is essential for the framework audit requirement under CSRD and IFRS S2.
