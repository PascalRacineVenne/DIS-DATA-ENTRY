Key implementation points
The template is the keystone
It bridges the abstract canonical model and the concrete DataSource. It encodes the columns required to satisfy all mapped frameworks. Versioned for backward compatibility.
canonical_id → template_id → DS_schema
Existing DataSource module stays unchanged
The DataSource module receives a schema_definition object generated by the template — exactly as if a user had manually configured the columns. Zero refactoring of the existing module.
template.to_schema() → DS.create(schema)
canonical_id as DataSource metadata
The created DataSource stores its canonical_id and template_id as metadata. This link allows the framework_renderer to know how to interpret and map columns to each framework.
DS.metadata.canonical_id = "GHG_SCOPE1…"
⚠ Design decisions to validate with leads
🔒
Locked template — what if a user needs a custom column?
Option A: Generic "notes" column for anything outside the template.
Option B: "Custom columns" section alongside template columns — not mapped to frameworks.
Option C: Template stays strictly locked, no exceptions.
🔄
Can an existing DataSource be retroactively linked to a canonical_id?
If an org already has manually created DataSources, do we need a migration workflow to attach them to a canonical_id? Or do we let free-form DSs and canonical DSs coexist?
📦
Can one canonical_id generate multiple DSs (multi-scope)?
E.g. Scope 1 by facility in one DS, Scope 1 aggregated at group level in another. Same canonical_id — but different granularities. How do we handle aggregation at render time?
🔢
Template versioning over time
If CSRD adds a new requirement in 2026 and template v1 is updated to v2, do DSs created on v1 remain valid? Do we need a data migration mechanism or inter-version comparison?
