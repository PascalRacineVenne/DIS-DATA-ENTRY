Three components — one for each team
1
The Canonical Model
A permanent, framework-agnostic list of indicators. Owned by the product team. Never changes when a new framework is added.
Each indicator has a unique ID (e.g. GHG_SCOPE1_DIRECT_tCO2e)
Labels are in plain business language — no ESRS, no C6.1
Includes domain (E/S/G), data type, unit, and taxonomy
This is what the user sees and searches through
→ Deep dive: ESG Canonical Model — Technical Architecture
2
The Mapping Table
The bridge between the canonical model and every framework. Adding a new framework means adding rows here — and nothing else.
One canonical ID → many framework codes (ESRS, GRI, CDP…)
Stores the official question label, section, and whether it's mandatory
Handles 3 match types: exact, partial, divergent
Framework renderer reads this table at export time
→ Deep dive: ESG Canonical Model — Technical Architecture
3
The DataSource Layer
The existing module — unchanged. The canonical model generates a pre-configured schema and passes it in. Users fill rows, not columns.
Template auto-creates required columns (year, facility, value…)
Schema is locked — users only enter data, not structure
Existing DSs can be retroactively linked to a canonical ID
DS stores canonical_id + template_id as metadata

Two things to know before diving deeper
🔗
Existing DataSources can be linked retroactively
If an organization already has manually created DataSources, they don't start from scratch. The system assesses the existing schema, auto-maps matching columns, flags missing ones, and adds only what's needed. Existing data and columns are never removed or renamed.
⚡
Adding a new framework costs almost nothing
Because the canonical model is the source of truth, adding a new reporting standard (e.g. Singapore Climate Reporting, SEC Climate Rule) means writing mapping rows — no schema changes, no new UI, no new data collection. Users who have already entered data see it appear automatically in the new framework's output.
